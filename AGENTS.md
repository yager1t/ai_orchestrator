# AGENTS.md

Инструкции для Codex и других coding agents в этом репозитории.

Уровень рассуждения:

- Medium — режим по умолчанию для обычной разработки.
- High — использовать для архитектурных решений, supervisor FSM, AgentAdapter, VerificationRunner, PolicyEngine, subprocess/PTY, resume/state store, security и сложной отладки.
- Low — использовать только для простых текстовых правок, переименований, README, changelog и очевидного boilerplate.

Если задача затрагивает безопасность, выполнение shell-команд, управление агентами, хранение состояния или принятие решений supervisor-ом — автоматически переходи на High.

## 1. Назначение проекта

Мы разрабатываем MVP `ai-orch` — локальный supervisor-оркестратор для CLI-агентов:

- Codex CLI;
- Claude Code;
- Gemini CLI;
- Kimi Code CLI;
- Generic CLI agents.

Система должна доводить пользовательскую задачу до результата через цикл:

```text
plan -> execute -> verify -> continue/done/blocked
```

Ключевой принцип: **агент-исполнитель не решает сам, что задача завершена**. Завершение подтверждает только supervisor после проверок.

---

## 2. Главные рабочие правила

1. Работай маленькими итерациями.
2. Перед изменением объясни, какой bounded step выполняешь.
3. Не переписывай архитектуру без отдельного решения в `docs/DECISIONS.md`.
4. Не меняй несвязанные файлы.
5. Не добавляй production-зависимости без обоснования.
6. Не удаляй файлы без явной причины.
7. Не трогай секреты, `.env`, ключи, токены, `~/.ssh`, `~/.codex/auth.json`.
8. Не делай `git push`, публикацию пакетов, деплой или destructive-команды без явного подтверждения пользователя.
9. После изменения кода запускай проверки.
10. Если проверки не проходят — исправляй минимально необходимым diff.
11. Если задача не может быть завершена — фиксируй blocker, причину и следующий ручной шаг.

---

## 3. Режимы работы

Всегда явно указывай режим:

### EXPLAIN

Только объяснение. Код не менять.

### PROPOSE

Предложить план, список файлов и риск. Код не менять.

### APPLY

Внести изменения по согласованному bounded step.

### VERIFY

Запустить проверки и описать результат.

### REPAIR

Исправить ошибку после проверки.

### REVIEW

Проверить diff, риски, архитектуру, тесты, безопасность.

Если пользователь не указал режим, по умолчанию используй `PROPOSE` для крупных изменений и `APPLY` для маленьких очевидных задач.

## 4. Архитектурные границы MVP

### В MVP

- CLI-команды.
- Минимальный Python package.
- Supervisor FSM.
- AgentAdapter interface.
- MockAgentAdapter.
- GenericCLIAdapter.
- VerificationRunner.
- SQLite state store.
- PolicyEngine.
- Markdown report.
- Конфигурация `.ai-orch/config.yaml`.

### Не в MVP

- Полноценная GUI/RPA-автоматизация окон.
- Полная поддержка MCP/ACP.
- Web UI.
- Cloud multi-user deployment.
- Автоматический git push.
- Автоматическое выполнение опасных команд.

## 6. Распределение агентских ролей

При работе мысленно разделяй задачи по ролям:

- `Supervisor Agent` — управляет циклом, статусами, DoD.
- `Architect Agent` — отвечает за границы модулей.
- `Core Agent` — реализует FSM, planner, decision engine.
- `Adapter Agent` — реализует Codex/Claude/Gemini/Kimi/Generic adapters.
- `Verification Agent` — тесты, lint, command runner.
- `Security Agent` — sandbox, approval, policy.
- `Docs Agent` — README, отчёты, changelog.
- `Review Agent` — проверяет diff перед финалом.

Не смешивай роли в одном большом изменении. Один PR/итерация — одна основная роль.

---

## 7. Definition of Done для любой задачи

Задача готова только если:

1. Изменения минимальны и соответствуют поставленной цели.
2. Код запускается.
3. Тесты проходят.
4. Есть проверка негативного сценария, если это логика ошибок.
5. Обновлена документация, если изменилось поведение.
6. Обновлён `CHANGELOG.md`, если это пользовательски заметное изменение.
7. Нет секретов, токенов, абсолютных путей пользователя.
8. Нет несанкционированных изменений за пределами задачи.

## 8. Обязательные проверки

Минимально:

```bash
python -m pytest
python -m compileall ai_orchestrator
```

Если добавлены форматтеры/линтеры, также запускать:

```bash
ruff check .
mypy ai_orchestrator
```

Если зависимости ещё не установлены, не выдумывай результат. Напиши, что проверка не была выполнена, и почему.

---

## 9. Работа с git

Перед крупным изменением:

```bash
git status --short
```

После изменения:

```bash
git diff --stat
git diff --check
```

Не коммить автоматически, если пользователь не просил. Но в отчёте предлагай название коммита.

Формат proposed commit message:

```text
feat(core): add supervisor state machine skeleton
fix(verification): handle command timeout
docs: add Codex workflow rules
```

## 10. Правила кода

- Python 3.12+.
- Предпочитать стандартную библиотеку, пока не нужна зависимость.
- Публичные структуры — typed dataclasses или Pydantic позднее.
- Ошибки возвращать явно через result/status objects.
- Не прятать исключения молча.
- Логика должна быть покрыта тестами.
- Shell-команды запускать только через отдельный runner.
- Все timeout-значения должны быть конфигурируемыми.
- Все внешние агенты подключаются только через `AgentAdapter`.

---

## 11. Правила безопасности

Запрещено без подтверждения:

- `rm -rf`;
- удаление за пределами repo;
- `git push`;
- публикация пакетов;
- изменение глобальных настроек системы;
- чтение приватных ключей;
- чтение `~/.codex/auth.json`;
- отправка содержимого проекта во внешние сервисы вне задачи;
- запуск неизвестных бинарников.

Все опасные действия должны идти через `PolicyEngine`.

---

## 12. Как отвечать пользователю после изменения

Всегда возвращай:

```text
Что сделано:
- ...

Проверки:
- команда: результат

Изменённые файлы:
- ...

Риски/ограничения:
- ...

Следующий шаг:
- ...
```

Не пиши “готово”, если проверки не пройдены.
